# VideoGrab 万能视频下载器 — 产品设计与开发报告

---

## 一、产品背景

### 1.1 研发动机

随着短视频和在线视频内容的爆发式增长，用户在日常学习、工作和娱乐中频繁产生「保存视频」的需求。然而，现有解决方案普遍存在以下痛点：

| 痛点类别 | 具体表现 | 影响用户 |
|---------|---------|---------|
| 平台分散 | B站、YouTube、抖音、优酷等平台各自为阵，用户需要为每个平台寻找不同的下载工具 | 所有用户 |
| 操作复杂 | 传统命令行工具（如 yt-dlp）需要技术背景，普通用户难以使用 | 非技术用户 |
| 画质受限 | 大多数在线工具只提供单一画质，无法满足创作者对高清素材的需求 | 视频创作者 |
| 内容理解困难 | 下载视频后，用户需要从头观看才能了解内容要点，时间成本高 | 学习者、研究者 |
| 字幕缺失 | 外语视频或无字幕视频难以理解，手动提取字幕耗时费力 | 学习者 |

基于以上痛点，我们决定开发 VideoGrab —— 一个集「视频下载 + AI 智能分析」于一体的在线平台，目标是让用户在一个工具内完成「解析 → 下载 → 理解」的完整流程。

### 1.2 目标用户

| 用户群体 | 核心需求 | 使用频率 |
|---------|---------|---------|
| 大学生 | 下载教学视频离线学习，AI 总结快速提取课程要点 | 每周 3-5 次 |
| 视频创作者 | 下载高清素材用于二次创作，提取字幕辅助文案 | 每天 1-3 次 |
| 研究人员 | 批量下载学术讲座视频，AI 提取关键论点 | 每周 1-2 次 |
| 普通用户 | 保存喜欢的视频到本地，离线观看 | 每周 1-2 次 |

### 1.3 预期价值

- **效率提升**：AI 视频总结功能可在 10 秒内将 1 小时视频的核心要点提炼出来，节省 90% 的观看时间
- **一站式体验**：下载 + 字幕 + AI 总结 + 思维导图，一个工具解决所有需求
- **降低门槛**：图形化界面，粘贴链接即可使用，无需任何技术背景
- **平台覆盖广**：基于 yt-dlp（14 万+ Star），支持 1800+ 视频平台

## 二、产品使用说明书

### 2.1 首页

首页是用户进入网站后看到的第一个页面，整体采用清爽极简白底设计。

**页面结构（从上到下）**：

1. **顶部导航栏** —— 固定在页面顶部，包含 Logo、导航链接（首页/功能介绍/工具对比/常见问题/下载历史）、Cookies 上传按钮、登录/注册按钮
2. **首屏 Hero 区域** —— 大标题「万能视频下载，一键搞定」+ 视频链接输入框 + 解析按钮 + 平台标签（YouTube/B站/抖音/Twitter 等）
3. **功能特性区** —— 四宫格卡片展示：全平台支持、高清画质、极速下载、安全隐私
4. **使用步骤区** —— 三步流程：粘贴链接 → 选择画质 → 一键下载
5. **定价区** —— 三栏展示：免费版（¥0）、Pro 版（¥19.9/月）、Premium 版（¥49.9/月）
6. **平台状态区** —— 实时展示各平台支持状态，绿/黄/橙/红四色指示灯
7. **页脚** —— Logo、版权信息、功能/FAQ/隐私政策等链接

**操作说明**：

用户在首屏输入框中粘贴视频链接（支持 YouTube、B站、抖音等任意平台），点击「解析视频」按钮，系统自动识别平台并解析视频信息。解析成功后弹出下载弹窗。

### 2.4 Cookies 与 SESSDATA 配置

**其他平台 Cookies**：
- 使用 Chrome 扩展「Get cookies.txt LOCALLY」导出 cookies.txt 文件

## 三、核心技术说明

### 3.1 大模型的具体运用方式

#### 3.1.1 模型选型

| 维度 | 选择 | 理由 |
|------|------|------|
| 模型 | DeepSeek API（deepseek-chat） | 中文能力强，性价比高，OpenAI 兼容 SDK |
| 接口协议 | OpenAI Chat Completions API | 流式输出（stream=True），逐 token 推送 |
| 上下文窗口 | 128K tokens | 足以容纳大部分视频的完整字幕 |
| temperature | 总结 0.3 / 问答 0.5 | 总结需要稳定性，问答需要一定创造性 |

#### 3.1.2 AI 视频总结流程

整个 AI 总结功能的数据流如下：

```
用户粘贴视频 URL
       │
       ▼
  获取视频字幕（四级降级策略）
       │
       ├─ 第 1 级：平台字幕 API
       │    B站：/x/v2/dm/view（不需要登录）
       │    YouTube：yt-dlp 自动获取
       │
       ├─ 第 2 级：平台自动字幕
       │    YouTube auto_captions（AI 生成）
       │
       ├─ 第 3 级：标题+描述兜底
       │    无字幕时使用视频元数据
       │
       ▼
  构建 Prompt
       │
       System Prompt（固定）：
       「你是一位专业的视频内容分析师。
        请根据视频信息和字幕内容，生成结构化的内容总结。
        输出格式：摘要 + 章节时间线 + 关键要点 + 标签 + 思维导图」
       │
       User Prompt（动态）：
       「请总结以下视频内容：
        视频标题：xxx
        字幕来源：平台字幕
        字幕/转录文本：[完整字幕内容]」
       │
       ▼
  调用 DeepSeek API（stream=True）
       │
       ▼
  SSE 流式返回 → 前端逐字渲染 Markdown
       │
       ▼
  自动提取 Mermaid 代码块 → 思维导图渲染
```

#### 3.1.3 Prompt 设计细节

**总结 Prompt**（System 角色）要求大模型输出固定结构：

```markdown
## 📝 内容摘要
（2-3 句话概括核心内容）

## 📑 章节时间线
- [MM:SS] 章节标题：简要描述

## 💡 关键要点
- 要点 1
- 要点 2
（3-7 个核心要点）

## 🏷️ 标签
标签1, 标签2, 标签3

## 🧠 思维导图
（Mermaid mindmap 语法）
```

这种结构化 Prompt 设计的好处是：输出格式稳定可预测，前端可以精确地将不同部分渲染到对应 UI 区域（摘要区域、时间线区域、导图区域）。

**问答 Prompt** 将完整字幕作为上下文注入 System 角色，用户问题作为 User 角色发送。支持多轮对话，最多保留最近 10 轮历史。

#### 3.1.4 B站字幕获取的技术突破

B站字幕获取是整个 AI 功能中最具技术挑战的部分。最初我们尝试了以下方案：

| 方案 | 结果 | 原因 |
|------|------|------|
| `/x/player/v2` API | ❌ 返回空数组 | 必须携带 SESSDATA（登录态） |
| `/x/player/wbi/v2` + WBI 签名 | ❌ 返回空数组 | WBI 签名正确但仍需登录态 |
| yt-dlp 字幕提取 | ❌ 412 错误 | B站反爬机制拦截 |
| 页面 HTML 抓取 | ❌ 无数据 | B站使用客户端渲染，HTML 中无字幕 |
| **`/x/v2/dm/view` API** | **✅ 成功** | **不需要 SESSDATA，始终返回字幕列表** |

最终发现的 `/x/v2/dm/view` API 是一个弹幕视图接口，其中包含了字幕列表信息，且不需要登录态即可访问。这是一个关键的技术突破，使得所有用户（无论是否登录 B站）都能获取字幕并使用 AI 总结功能。

### 3.2 AI Coding 过程中的关键问题与解决方法

本项目全程使用 AI Coding 工具（Claude Code）辅助开发。以下是开发过程中遇到的关键问题、解决方法和反思。

#### 问题 1：B站 412 反爬 —— AI 无法预见平台级反爬升级

**问题描述**：
项目最初使用 yt-dlp 统一处理所有平台。开发过程中发现 B站视频解析始终返回 `HTTP Error 412: Precondition Failed`。AI 编码工具按照 yt-dlp 文档生成的代码完全正确，但无法解决 B站 2024 年新启用的 WBI 签名反爬机制。

**解决过程**：
1. AI 首先尝试添加 cookies、自定义 User-Agent、Referer 头 → 仍然 412
2. AI 尝试使用 yt-dlp 的 `cookiesfrombrowser` 参数 → 仍然 412
3. **转折点**：通过搜索发现 B站 API 的 buvid 指纹机制，AI 改为直接调用 B站内部 API
4. 最终方案：完全绕过 yt-dlp，自建 B站 API 处理器（`bilibili.py`），通过 `/x/frontend/finger/spi` 获取 buvid 指纹 + `/x/web-interface/view` 获取视频信息 + `/x/player/wbi/playurl` 获取流地址

**反思**：
AI 编码工具擅长根据已知文档生成代码，但面对平台级的反爬策略升级时，需要人工介入提供「绕过」思路。这说明 AI Coding 更适合作为「执行者」而非「架构决策者」，关键的技术路线判断仍需人类经验。

---

#### 问题 2：YouTube 下载没声音 —— AI 默认配置与实际格式不匹配

**问题描述**：
YouTube 视频使用 DASH 格式（视频流和音频流分开存储）。AI 生成的 yt-dlp 配置只指定了 `format: "best"`，导致只下载了视频流而没有音频流，下载的 MP4 文件没有声音。

**解决过程**：
1. AI 首先尝试 `format: "bestvideo+bestaudio"` → 部分视频仍然无声（VP9+Opus 编码不兼容）
2. 最终方案：`format: "bestvideo[vcodec^=avc1]+bestaudio[acodec^=mp4a]/bestvideo+bestaudio/best"` + `format_sort: ["vcodec:h264", "acodec:aac"]`
3. 强制选择 H.264 视频 + AAC 音频，保证所有播放器兼容

**反思**：
AI 在生成 yt-dlp 配置时倾向于使用「看起来正确」的通用参数，但忽略了不同平台的编码差异。这类问题需要实际测试才能发现，仅靠代码审查无法暴露。

---

#### 问题 3：前端错误显示 `[object Object]` —— AI 生成的错误处理不够健壮

**问题描述**：
下载失败时，前端错误信息显示为 `[object Object]` 而非可读文字。原因是后端 Pydantic 验证错误返回的是数组格式的 `detail` 字段，而 AI 生成的错误处理代码只处理了字符串格式。

**解决过程**：
AI 生成了 `safeStr()` 工具函数，统一处理多种错误类型：
```javascript
function safeStr(val) {
  if (!val) return ''
  if (typeof val === 'string') return val
  if (val instanceof Error) return val.message
  if (typeof val === 'object') 
    return val.message || val.detail || val.error || JSON.stringify(val)
  return String(val)
}
```

**反思**：
AI 在生成代码时倾向于处理「正常路径」，对异常路径（如 Pydantic 返回数组、SSE 返回对象）的处理不够全面。开发者需要主动构造异常场景进行测试，而不是信任 AI 生成的「看起来完整」的错误处理。

---

#### 问题 4：B站字幕获取 —— AI 反复尝试已知方案，缺乏创造性突破

**问题描述**：
B站字幕获取是整个项目中 AI Coding 表现最弱的环节。AI 在已知 API（`/x/player/v2`）上反复尝试不同的认证方式（SESSDATA、WBI 签名、buvid 指纹），但都无法突破「必须登录才能获取字幕」的限制。

**解决过程**：
1. AI 尝试 `/x/player/v2` + 无认证 → 空数组
2. AI 尝试 `/x/player/v2` + SESSDATA → 成功（但要求用户登录）
3. AI 尝试 `/x/player/wbi/v2` + WBI 签名 → 仍然空数组
4. AI 尝试 yt-dlp 字幕提取 → 412 错误
5. AI 尝试页面 HTML 抓取 → 无数据（客户端渲染）
6. **人工介入**：通过搜索 B站开发者社区，发现 `/x/v2/dm/view` 这个弹幕视图 API 不需要登录即可返回字幕列表
7. 将发现告诉 AI 后，AI 迅速完成了双 API 策略的代码实现

**反思**：
这是 AI Coding 最明显的局限性 —— AI 只能在「已知知识」范围内搜索解决方案，无法像人类一样通过社区交流、逆向工程等方式发现未文档化的 API。但一旦人类提供了方向性指导，AI 的执行效率极高（从「发现 API」到「完成双 API 策略代码」仅需几分钟）。

---

#### 问题 5：芒果TV 进度条一直为 0 —— AI 忽略了 HLS 下载的特殊性

**问题描述**：
芒果TV 使用 HLS（m3u8）格式传输视频，yt-dlp 下载 HLS 时使用分片进度（`fragment_index/fragment_count`）而非字节进度（`downloaded_bytes/total_bytes`）。AI 生成的进度回调只处理了字节进度，导致 HLS 下载时进度条始终为 0%。

**解决过程**：
AI 在被告知问题后，迅速扩展了进度回调函数，增加了三种进度计算方式的优先级：
```python
# 优先用字节进度
if total and downloaded:
    progress = downloaded / total * 100
# 其次用 HLS 分片进度
elif d.get("fragment_index") and d.get("fragment_count"):
    progress = d["fragment_index"] / d["fragment_count"] * 100
# 最后用百分比字符串回退
elif d.get("_percent_str"):
    progress = float(d["_percent_str"].replace("%", "").strip())
```

**反思**：
AI 对 yt-dlp 的 `progress_hooks` 回调数据结构了解不够深入，只覆盖了最常见的字节进度场景。这类「边界条件」问题需要通过实际测试不同平台才能发现。

---

#### 问题 6：抖音下载路由遗漏 normalize_url —— AI 修改时的「遗漏链」

**问题描述**：
抖音精选页链接（`jingxuan?modal_id=xxx`）在解析时能正常工作，但下载时报错「不支持的链接格式」。原因是解析路由已经调用了 `normalize_url()` 进行 URL 规范化，但下载路由中遗漏了这一步 —— 仍然使用原始的 `req.url` 而非规范化后的 `url`。

**解决过程**：
一行代码修复：`start_download(req.url, ...)` → `start_download(url, ...)`

**反思**：
AI 在修改代码时容易出现「修了 A 忘了 B」的遗漏。当一个函数在多处被调用时，AI 可能只修改了当前上下文看到的调用点，而遗漏了其他调用点。这类问题的根源是 AI 的「局部视野」—— 它只能看到当前对话中被提及的代码，而非整个代码库的全局视图。

### 3.3 AI Coding 总结与反思

#### AI Coding 的优势

| 优势 | 具体表现 |
|------|---------|
| **快速原型** | 从零搭建 Vue 3 + FastAPI 全栈项目仅需几分钟，包括路由、状态管理、API 封装等基础设施 |
| **代码质量** | 生成的代码风格一致、注释规范、结构清晰，符合最佳实践 |
| **文档能力** | 自动生成架构图、API 文档、问题记录等，质量接近人工编写的水平 |
| **重构效率** | 从深色 UI 重构为白底极简风格，涉及 12 个组件的样式修改，AI 在 1 小时内完成 |
| **错误修复** | 大部分 bug（如 ffmpeg 路径、os 模块导入、Pydantic 类型转换）AI 能快速定位并修复 |

#### AI Coding 的局限

| 局限 | 具体表现 | 应对策略 |
|------|---------|---------|
| **无法预见平台变化** | B站 412 反爬、爱奇艺 API 变更等需要人工调研 | 人类提供方向，AI 负责实现 |
| **边界条件覆盖不足** | HLS 分片进度、Pydantic 数组错误等非典型场景被遗漏 | 人工构造异常测试用例 |
| **缺乏全局视野** | 修改 A 函数时遗漏 B 函数中的相同调用 | 使用全局搜索验证修改完整性 |
| **创造性不足** | 无法发现未文档化的 API（如 `/x/v2/dm/view`） | 人类提供线索，AI 快速实现 |
| **过度自信** | 生成的代码「看起来正确」但未经测试验证 | 每个功能必须实际运行测试 |

#### 关键经验

> **最佳协作模式**：人类负责「做什么」和「为什么」，AI 负责「怎么做」。人类提供技术方向和业务判断，AI 负责代码实现和文档编写。这种分工下，开发效率提升约 3-5 倍，但人类的监督和测试不可省略。
